# SafetyPulse 面试介绍话术

## 30 秒版本

我做了一个叫 SafetyPulse 的平台安全智能分析台，模拟 TikTok Trust & Safety 场景下的数据分析产品。它把内容安全、直播安全、账户安全、隐私合规和审核平台的指标接入后，自动形成核心指标体系，识别异常，展示趋势和预测，用柱状图比较业务域、地区和政策风险，下钻到高风险账号样本，并把分析结论转成治理动作。这个项目想展示的不是单纯 dashboard，而是从指标定义、异常检测、根因分析、case drill-down 到业务闭环的一套可复用分析系统。

## 2 分钟版本

这个尝试是根据 JD 里几个关键词设计的：核心指标体系、异常监控、风险事件分析、数据挖掘、跨团队推动改进，以及把分析能力产品化。

我把系统分成四层：

1. **指标层**：定义平台安全核心指标，比如举报率、违规命中率、审核耗时、申诉成功率、高风险账号和安全事件数。指标支持按业务域、地区、语言和政策类型切片。
2. **检测层**：用过去一段时间的历史窗口作为 baseline，对最新数据做异常检测，并设置最小样本门槛，避免低基数噪声造成误报。异常会被分成 P0、P1、P2。
3. **诊断层**：对异常指标做贡献度拆解，比如发现整体违规率上升主要来自 `Live / ID / Harassment`，这样业务团队可以快速定位问题而不是只看到一个全局数字。
4. **样本层**：把宏观异常下钻到高风险账号，展示风险分、账号簇、举报量、违规量、设备数和处置建议，方便产品、运营和审核团队直接拿样本复盘。
5. **行动层**：把异常转成治理动作，比如调整高风险直播间分流、补充审核人力、更新风险评分特征、启动阈值实验，并记录 owner、状态和预期影响。

从技术上，我用 Python 标准库和 SQLite 做了一个轻量后端，用原生前端和 SVG 做系统式工作台。界面不是单页报表，而是左侧导航的多工作区：总览、图表分析、异常诊断、风险账号和治理闭环。数据是合成的，但刻意模拟了真实 Trust & Safety 场景中的波动，比如直播骚扰举报上升、账号诈骗风险上升、隐私误伤申诉上升和审核 SLA 积压。

## 面试官可能会追问

### 你为什么做这个项目？

我看到 JD 里强调的不只是写 SQL 或做报表，而是要针对安全业务挑战搭建定量分析系统，并把洞察推动成实际改进。所以我想做一个能体现完整闭环的项目：从数据接入到异常发现，再到根因定位和治理动作。

### 这个系统解决什么业务问题？

它解决的是安全团队面对海量指标时的三个痛点：

- 第一，核心指标分散，难以及时看到平台安全状态。
- 第二，异常发生后定位慢，通常需要人工反复切维度。
- 第三，分析结论容易停留在报告里，没有变成 owner 明确的行动项。

SafetyPulse 把这三件事串起来，让安全运营、产品、算法和审核团队能共享同一套事实。

### 异常检测是怎么做的？

MVP 里我用了比较透明的统计方法：对每个业务切片建立历史 baseline，计算最新值相对 baseline 的变化和波动分数，再结合最小样本量门槛过滤低基数误报。这个方法可解释性强，适合安全运营落地。后续可以升级为季节性模型、EWMA、Prophet 或 Isolation Forest。

### 根因分析怎么做？

我把指标按 `业务域 / 地区 / 政策类型` 聚合，比较当前窗口和历史窗口的增量，然后计算每个切片对正向异常增量的贡献度。这样可以回答“为什么全局指标上升”，比如是直播印尼骚扰问题拉动，还是巴西账号诈骗拉动。

### 为什么要加高风险账号明细？

安全分析最终要落到可复盘的 case。只看指标和异常还不够，业务团队需要知道“哪些账号、哪个账号簇、什么行为特征”在拉高风险。所以我加了账号 drill-down，把异常切片连接到具体账号样本，展示风险分、举报/违规量、设备数、账号簇和建议动作。这样分析结论可以直接交给审核、策略或算法团队使用。

### 趋势和预测怎么做？

MVP 里用了轻量、可解释的方法：展示实际值、7 天 rolling baseline，并基于最近 14 天做线性趋势预测，同时给出简单波动区间。这个预测不是为了替代复杂模型，而是帮助业务提前判断风险是否可能继续扩大。后续可以升级成季节性模型、Prophet、EWMA 或带业务日历特征的时序模型。

### 这个项目如何与算法团队合作？

当异常和根因定位到某类高风险切片后，可以把样本和特征方向同步给算法团队。例如账号诈骗异常可以进一步看设备聚类、注册路径、行为序列特征，然后把高增益特征加入风险评分模型。分析系统负责发现问题、量化影响和定义实验护栏。

### 这个项目还有哪些可以改进？

后续我会做三类升级：

- 数据层：接真实埋点、审核日志、举报日志和模型结果表。
- 方法层：加入季节性异常检测、因果影响评估和自动 case mining。
- 产品层：支持告警订阅、样本 drill-down、实验追踪和治理效果回看。

## 一句话收尾

这个项目体现的是我对 Trust & Safety 数据工作的理解：安全数据分析不只是发现异常，而是要把异常变成可以定位、可以协作、可以验证效果的治理闭环。
